# Hide Email Extension — Implementation Plan

A cross-browser extension built with [WXT](https://wxt.dev) that scans every web page for a user-defined list of email addresses and replaces each occurrence with `[Email Redacted]`. Written in 100% TypeScript with strict type safety and minimal dependencies.

---

## 1. Goals & Non-Goals

### Goals

- Redact any email address from a user-managed allow/block list on **every** web page.
- Display the literal token `[Email Redacted]` in place of the matched address.
- Provide a small **popup UI** (no framework — plain TS + DOM) to add/remove emails.
- Persist the email list across browser sessions and sync across devices when available (`browser.storage.sync`).
- Re-run redaction on dynamic DOM changes (SPA navigation, infinite scroll, late-loaded content).
- Be fully type-safe end-to-end (storage, messaging, DOM walking).
- Work on **all pages** (`<all_urls>`) on Chromium, Firefox, and (best-effort) Safari via WXT.

### Non-Goals (v1)

- No regex-based "all emails" detection — only the user-supplied list is redacted.
- No password/PII detection beyond emails.
- No options page (popup is sufficient for v1).
- No framework (React/Vue/Svelte) — keeps bundle and deps minimal.
- No remote sync server — relies on browser-built-in sync only.

## 4. Type Model (`src/lib/types.ts`)

```ts
/** A normalized (trimmed, lower-cased) email address. Branded for safety. */
export type NormalizedEmail = string & { readonly __brand: "NormalizedEmail" };

/** Shape persisted in browser.storage.sync. */
export interface StorageSchema {
  hiddenEmails: NormalizedEmail[];
}

/** Runtime messages — discriminated union for exhaustive handling. */
export type Message =
  | { type: "EMAILS_UPDATED"; emails: NormalizedEmail[] }
  | { type: "REQUEST_RESCAN" };

/** Replacement token rendered in place of each match. */
export const REDACTION_TOKEN = "[Email Redacted]" as const;
export type RedactionToken = typeof REDACTION_TOKEN;
```

All other modules import from this file so the schema is single-sourced.

## 5. Storage Layer (`src/lib/storage.ts`)

Uses WXT's typed [`storage`](https://wxt.dev/storage.html) helper, which gives compile-time keys + values.

```ts
import { storage } from "#imports"; // WXT auto-import
import type { NormalizedEmail } from "./types";

export const hiddenEmailsItem = storage.defineItem<NormalizedEmail[]>(
  "sync:hiddenEmails",
  { fallback: [] },
);

export async function getHiddenEmails(): Promise<NormalizedEmail[]> { … }
export async function addHiddenEmail(email: NormalizedEmail): Promise<void> { … }
export async function removeHiddenEmail(email: NormalizedEmail): Promise<void> { … }
export function watchHiddenEmails(
  cb: (next: NormalizedEmail[]) => void,
): () => void { … } // returns unwatch fn
```

- Storage area: `sync` so the list follows the user across devices; falls back to `local` automatically on browsers without sync.
- All writes deduplicate via a `Set`.

## 6. Email Validation (`src/lib/email.ts`)

```ts
import type { NormalizedEmail } from "./types";

const EMAIL_RE = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

export function isEmail(value: unknown): value is string {
  return typeof value === "string" && EMAIL_RE.test(value.trim());
}

export function normalize(email: string): NormalizedEmail {
  return email.trim().toLowerCase() as NormalizedEmail;
}
```

A single tiny regex — good enough for the user's own list (they wouldn't enter junk on purpose), and we don't try to parse RFC 5322.

## 7. Redaction Engine (`src/lib/redact.ts`)

Pure, DOM-only module — no `browser.*` calls. Easy to unit test.

### Algorithm

1. Build a single `RegExp` from the email list: `new RegExp(emails.map(escape).join("|"), "gi")`.
   - If the list is empty, do nothing.
   - Cache the compiled regex; only rebuild when the list changes.
2. Walk the DOM with `document.createTreeWalker(root, NodeFilter.SHOW_TEXT)`, skipping:
   - Nodes inside `<script>`, `<style>`, `<noscript>`, `<textarea>`, `<input>`, `contenteditable` elements.
   - Nodes the extension itself created (tagged with a `data-redacted` ancestor).
3. For each text node whose `nodeValue` matches: replace with `nodeValue.replace(regex, REDACTION_TOKEN)`.
4. Also scan attributes that commonly render to UI: `title`, `aria-label`, `alt`, `placeholder`, `value` (only for non-editable inputs).
5. **MutationObserver** on `document.documentElement` with `{ childList: true, subtree: true, characterData: true, attributes: true, attributeFilter: [...attrs] }` re-runs step 2–4 on changed sub-trees only (not the whole doc) — debounced via `requestIdleCallback` (with `setTimeout` fallback) to coalesce bursts.
6. Re-listen to storage changes so updates from the popup take effect immediately without a page reload.

### Public surface

```ts
export interface Redactor {
  setEmails(emails: NormalizedEmail[]): void;
  start(): void; // attach observer + initial scan
  stop(): void;
}

export function createRedactor(root?: Document): Redactor;
```

### Edge cases handled

- Case-insensitive match (`gi` flag + normalization on input).
- Special regex chars in emails escaped (`+`, `.`).
- Shadow DOM: best-effort — walk open shadow roots discovered during traversal.
- Iframes: each frame gets its own content script via WXT manifest `all_frames: true`.
- Input fields: **never** mutate user-editable values (would break forms).

## 8. Content Script (`entrypoints/content.ts`)

```ts
import { defineContentScript } from "wxt/sandbox";
import { createRedactor } from "@/lib/redact";
import { hiddenEmailsItem, watchHiddenEmails } from "@/lib/storage";

export default defineContentScript({
  matches: ["<all_urls>"],
  allFrames: true,
  runAt: "document_start", // start observer before page renders; initial scan on DOMContentLoaded
  async main() {
    const redactor = createRedactor();
    redactor.setEmails(await hiddenEmailsItem.getValue());
    redactor.start();
    watchHiddenEmails((next) => redactor.setEmails(next));
  },
});
```

- `runAt: document_start` minimizes the window in which emails are visible.
- Observer is wired up before paint so most replacements happen pre-render.

## 10. Popup UI (`entrypoints/popup/`)

Plain HTML + TS — no framework. Standard browser extension popup feel — compact, dark inputs, no decoration.

### Design principles

- **Standard popup**: ~300px wide, minimal height. Looks like a native browser extension popup — no splash, no hero, no logo.
- **Dark inputs**: The `<input>` and list area use a dark background (`#1e1e1e`–`#252525`) with light text so they blend with dark-mode browser themes. The overall popup background respects the browser's color scheme.
- **No custom fonts, no icons, no rounded cards** — just native HTML elements with subtle styling.
- **Respects `prefers-color-scheme`** — dark input styling is the default; on `prefers-color-scheme: light`, the popup stays compact but inputs remain dark (consistent feel).

### `index.html`

- `<input type="email" id="email-input">` + Add button.
- `<ul id="email-list">` showing the current list with a Remove (×) button per row.
- Inline error span for invalid input.

### `popup.ts`

- On load: render list from `hiddenEmailsItem.getValue()`.
- Add handler:
  1. Validate via `isEmail`, show inline error if not.
  2. `normalize` → `addHiddenEmail`.
  3. Re-render list.
- Remove handler: `removeHiddenEmail(email)` → re-render.
- Subscribe to `hiddenEmailsItem.watch` so the UI stays consistent if changed from another tab/device.

All DOM lookups go through small typed helpers (`getById<T>`) — no `as` casts elsewhere.

## 12. WXT Configuration (`wxt.config.ts`)

```ts
import { readFileSync, existsSync } from "node:fs";
import { resolve } from "node:path";
import { defineConfig } from "wxt";
import pkg from "./package.json";

function loadKey(): string | undefined {
  const keyPath = resolve(import.meta.dir, "key.pem");
  if (!existsSync(keyPath)) return undefined;
  return readFileSync(keyPath, "utf-8");
}

const key = loadKey();
if (!key) {
  console.warn(
    "[wxt] key.pem not found — extension ID will be random. Run `bun run generate-key`."
  );
}

export default defineConfig({
  manifest: {
    name: "Hide Email",
    description: "Redacts user-specified email addresses on every page.",
    version: pkg.version,
    permissions: ["storage"],
    host_permissions: ["<all_urls>"],
    action: { default_popup: "popup/index.html" },
    ...(key ? { key } : {}),
  },
  srcDir: ".",
});
```

- **Only the `storage` permission** is required — no `tabs`, no `scripting`, no `activeTab`. Smaller permission prompt at install.
- `<all_urls>` host permission is required so the content script can run everywhere.

## 17. Type-Safety Guarantees Summary

| Boundary               | How it's typed                                                                      |
| ---------------------- | ----------------------------------------------------------------------------------- |
| Storage read/write     | `storage.defineItem<NormalizedEmail[]>` — value type inferred everywhere.           |
| User input → email     | `isEmail` type guard + `normalize` returns branded `NormalizedEmail`.               |
| Runtime messages       | Discriminated union `Message`, validated by a guard before dispatch.                |
| DOM lookups in popup   | `getById<T extends HTMLElement>(id, ctor)` — narrows or throws, no casts.           |
| WXT manifest           | `defineConfig` from `wxt` — checked against MV3 schema.                             |
| Content-script options | `defineContentScript` — typed `matches`, `runAt`, etc.                              |
| `tsconfig`             | `strict`, `noUncheckedIndexedAccess`, `verbatimModuleSyntax`, `noImplicitOverride`. |

No `any`, no non-null assertions outside generated WXT code, no unchecked casts.

## 18. Persistent Extension ID via `key.pem`

Chrome assigns a random extension ID on every `wxt build` unless a cryptographic key is embedded in the manifest. A stable ID is required for:

- Consistent extension ID across dev/prod builds (so `storage.sync` data survives reinstalls).
- Chrome Web Store recognises the extension as the same listing.
- Content-script matching rules (`<all_urls>`) stay stable.

### Approach

1. A **`key.pem`** file (RSA private key) is generated once via a Bun script and stored **outside the repo** (never committed — it's in `.gitignore`).
2. The public key is derived from `key.pem` and injected into the manifest at build time via WXT's `manifest` hook, so the extension ID is deterministic.
3. Each developer generates their own `key.pem` locally (or shares it via a secure channel for team consistency).
4. CI pulls the key from a **GitHub Actions secret** (`EXTENSION_KEY_PEM`) so builds are reproducible.

### `scripts/generate-key.ts`

A small Bun script that generates a 2048-bit RSA key pair, writes the PEM to `key.pem` (gitignored), and prints the resulting extension ID so the developer can verify.

```ts
import { generateKeyPairSync } from "node:crypto";
import { writeFileSync } from "node:fs";
import { resolve } from "node:path";

const outPath = resolve(import.meta.dir, "..", "key.pem");

const { privateKey } = generateKeyPairSync("rsa", {
  modulusLength: 2048,
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});

writeFileSync(outPath, privateKey, { mode: 0o600 });
console.log(`key.pem written to ${outPath}`);
console.log("Add this file to .gitignore — it must never be committed.");
```

Run: `bun run generate-key`

### WXT config integration (`wxt.config.ts`)

```ts
import { readFileSync, existsSync } from "node:fs";
import { resolve } from "node:path";
import { defineConfig } from "wxt";

function loadKey(): string | undefined {
  const keyPath = resolve(import.meta.dir, "key.pem");
  if (!existsSync(keyPath)) return undefined;

}

const key = loadKey();

export default defineConfig({
  manifest: {
    name: "Hide Email",
    description: "Redacts user-specified email addresses on every page.",
    version: "0.1.0",
    permissions: ["storage"],
    host_permissions: ["<all_urls>"],

```

- If `key.pem` is absent (e.g., CI misconfigured), the build still works but gets a random ID — a warning is logged.
- The key is read at build time only; it is **not** bundled into the extension payload.

## 19. Versioning with Changesets

[Changesets](https://github.com/changesets/changesets) manages version bumps and changelogs. Every meaningful change gets a changeset file describing the bump type and a human-readable summary.

### Setup

1. `bun add -d @changesets/cli`
2. `bunx changeset init` — creates `.changeset/` directory and `config.json`.
3. Add scripts to `package.json`:

```jsonc
{
  "scripts": {
    // ...existing scripts...
    "changeset": "changeset",
    "version": "changeset version",
    "release": "changeset publish",
  },
}
```

Note: `release` no longer calls `bun run build` — the CI workflow handles building + zipping before invoking `changeset publish`, so the `release` script is purely the Changesets publish step.

### `.changeset/config.json`

```jsonc
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.1/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": [],
}
```

- `access: "restricted"` — this is a browser extension, not an npm package, but Changesets still needs this config. We use it purely for version tracking.
- The version in `package.json` is the single source of truth and is synced to the manifest `version` field at build time.

### Syncing `package.json` version → manifest

In `wxt.config.ts`, read the version from `package.json` so it's never duplicated:

```ts
import pkg from "./package.json";

export default defineConfig({
  manifest: {
    // ...
    version: pkg.version,
  },
});
```

### Workflow

1. Developer makes a change → runs `bunx changeset` → picks `patch`/`minor`/`major` → writes summary.
2. The changeset file (markdown + yaml frontmatter) is committed alongside the code change.
3. `bunx changeset version` consumes all pending changesets, bumps `package.json`, and updates `CHANGELOG.md`.
4. CI (see §20) automates the version PR.

## 20. CI — GitHub Actions

### Workflow: Version & Release

The workflow has two paths:

1. **No pending changesets** → does nothing (or creates a version PR if there are pending changesets).
2. **Changesets publishes** → builds Chrome + Firefox zips and creates a GitHub Release with both attached.

`.github/workflows/release.yml`:

```yaml
name: Release

on:
  push:
    branches: [main]

concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    name: Version or Publish
    runs-on: ubuntu-latest
    outputs:
      published: ${{ steps.changesets.outputs.published }}
      version: ${{ steps.version.outputs.version }}
    steps:
      - uses: actions/checkout@v4

      - uses: oven-sh/setup-bun@v2

      - run: bun install --frozen-lockfile

      - name: Write key.pem
        env:
          EXTENSION_KEY_PEM: ${{ secrets.EXTENSION_KEY_PEM }}
        run: echo "$EXTENSION_KEY_PEM" > key.pem

      - name: Create Release Pull Request or Publish
        id: changesets
        uses: changesets/action@v1
        with:
          publish: bun run release
          version: bun run version
          title: "chore: version extension"
          commit: "chore: version extension"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract published version
        if: steps.changesets.outputs.published == 'true'
        id: version
        run: echo "version=$(jq -r .version package.json)" >> "$GITHUB_OUTPUT"

  build-and-release:
    name: Build & Ship Zips
    needs: release
    if: needs.release.outputs.published == 'true'
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4

      - uses: oven-sh/setup-bun@v2

      - run: bun install --frozen-lockfile

      - name: Write key.pem
        env:
          EXTENSION_KEY_PEM: ${{ secrets.EXTENSION_KEY_PEM }}
        run: echo "$EXTENSION_KEY_PEM" > key.pem

      - name: Build Chrome + Firefox
        run: |
          bun run build
          bun run build:firefox

      - name: Generate zips
        run: |
          bun run zip
          bun run zip:firefox

      - name: Rename zips for clarity
        run: |
          chrome_zip=$(ls .output/*-chrome.zip 2>/dev/null | head -1)
          firefox_zip=$(ls .output/*-firefox.zip 2>/dev/null | head -1)
          if [ -z "$chrome_zip" ]; then echo "Chrome zip not found" && exit 1; fi
          if [ -z "$firefox_zip" ]; then echo "Firefox zip not found" && exit 1; fi
          mv "$chrome_zip" .output/hide-email-chrome.zip
          mv "$firefox_zip" .output/hide-email-firefox.zip

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          tag_name: v${{ needs.release.outputs.version }}
          name: v${{ needs.release.outputs.version }}
          generate_release_notes: true
          files: |
            .output/hide-email-chrome.zip
            .output/hide-email-firefox.zip
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### What happens on each merge to `main`

| Scenario                 | Result                                                                                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| No pending changesets    | Workflow runs, no-op.                                                                                                                                         |
| Pending changesets exist | Changesets action creates/updates a "Version extension" PR. When that PR is merged, the next run publishes the version and triggers `build-and-release`.      |
| Version just published   | `build-and-release` job runs: builds Chrome + Firefox, zips both, creates a GitHub Release (`vX.Y.Z`) with both zips attached + auto-generated release notes. |

### Workflow: CI Checks

`.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]

    branches: [main]

jobs:
  check:
    name: Typecheck & Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: oven-sh/setup-bun@v2

      - run: bun install --frozen-lockfile

      - run: bun run compile

      - run: bun run check
```
